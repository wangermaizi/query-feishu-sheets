# 执行状态机

## 状态

| 状态 | 含义 | 后续动作 |
| --- | --- | --- |
| `discovered` | 已查询，尚未选择 | 参与本轮排序 |
| `awaiting_analysis_confirmation` | 已完成批量分析，等待用户统一确认 | 不写最终结论、不修改代码 |
| `approved` | 用户已确认需要实施 | 按批次排序等待执行 |
| `needs_confirmation` | 必要性证据冲突或不足 | 等待用户判断，不选择新需求 |
| `awaiting_branch_choice` | 已选中，等待用户决定分支 | 禁止修改代码 |
| `in_progress` | 用户已授权工作位置 | 实施、测试和 Review |
| `report_failed` | 实施已完成但群发送失败 | 人工决定是否重发 |
| `reported` | 最终结果已发到群 | 后续查询排除 |
| `completed` | 代码核验确认已经实现 | 后续查询排除 |
| `obsolete` | 需求已失效或没有继续必要 | 后续查询排除 |
| `duplicate` | 与已有实现或需求重复 | 后续查询排除 |
| `skipped` | 用户取消或明确跳过 | 后续查询默认排除 |

状态转换只能前进，不要因为新一轮定时任务把未完成项重置为 `discovered`。

状态以仓库范围和需求 ID 共同作为内部键，并保留用户可见的 `requirement_id`。不同仓库允许并行；同一 worktree 禁止并行；同一 Git 仓库的不同 worktree 仅在用户明确确认后允许并行。状态修改必须通过 `run_state.py`，不得直接编辑 `state.json`，以保留进程锁和原子写入。

`in_progress` 内部使用 Review 阶段门禁：`implementation` → `review_ready` → `reviewing` → `review_findings_ready` → `review_complete`。需要严格档定向复审时，在集中修复后进入 `review_fixes`，重新准备完整修复候选，再进入第二轮。Reviewer 替换后再次超时、失败或退出则从 `reviewing` 进入终止等待的 `review_blocked`。不得跳过阶段、实现前启动 Reviewer，或在 Reviewer 尚未全部返回时修改代码。

## 每日触发

1. 检查模型能力和配置。
2. 查询需求并读取本地状态。
3. 按目标仓库筛选状态。若当前仓库存在同一 `batch_id` 的 `awaiting_analysis_confirmation`，重新展示完整批次并等待统一确认，不分析该仓库的新需求。
4. 对当前仓库运行 `check-scope`。同一 worktree 的 `awaiting_branch_choice`、`in_progress` 或 `report_failed` 才阻塞；不同仓库的活动需求只提示并继续。
5. 同一仓库不同 worktree 需要用户明确确认并行；旧活动记录缺少范围时使用 `attach-scope` 补齐，不得标记为 `skipped` 来解除全局锁。
6. 若当前仓库存在 `approved`，重新核对排名最高项的证据；结论未变化则进入分支确认。其他仓库的 `approved` 留给对应任务。
7. 否则分析并排序候选，对本轮所有候选完成只读必要性核验。
8. 把全部建议结论和普通语言需求说明写入同一待确认批次，一次性展示并暂停。每条需求同时展示原始内容、普通语言说明和技术分析。
9. 用户统一确认后写入终态或 `approved`，再选择排名最高的已批准需求。
10. 主 Agent 对选中需求运行复杂度分级，把分级与验证策略放入分支确认；不得未分级直接实施。

## 手动触发

1. 用户直接描述具体需求时选择手动入口，不查询飞书，不要求 profile 或筛选条件。
2. 从本次消息提取需求；可概括标题，并为未明确提供的验收标准生成可验证草稿。
3. 使用已确认默认值补齐项目名和仓库；无法唯一确定时询问用户。
4. 运行 `manual_requirement.py` 校验、检查完全重复内容并生成本地 ID。
5. 对全部手动需求执行只读必要性核验，与普通语言说明及验收标准草稿一起批量展示。
6. 用户统一确认后进入与飞书来源完全相同的状态机，本轮仍最多实施一条，并在分支确认前由主 Agent 完成复杂度分级。

恢复手动需求时从 `state.json` 读取标题、原始描述、验收标准、项目名、仓库、来源和内容指纹。不得因为没有飞书记录而重新生成 ID 或丢失原始内容。

## 必要性核验

逐项对照验收标准并保存证据。`partially_done` 必须明确剩余范围；后续实施与 Review 只针对剩余范围。不要为了完成核验而修改文件、安装依赖或切换分支。必须完成整个候选批次后再请求一次统一确认，不得因某条 `needs_confirmation` 提前中断其余分析。

首次分析每条需求时同步生成普通语言说明，依次覆盖当前情况、问题或用户影响、期望结果；复杂场景补充简单例子。批量确认页必须让用户在同一条需求下同时看到这段说明与技术结论、证据和剩余验收项，恢复等待中的批次时也必须完整重现。

## 用户回复后恢复

把回复与等待中的需求 ID 对应，并恢复已保存的复杂度分级与验证策略。用户选择当前分支时不得创建分支；用户选择新分支时使用 `<type>/<YYYYMMDD>/<summary>` 格式并在创建前确认完整名称。类型只能为 `feature`、`fix` 或 `refactor`，日期使用 Asia/Shanghai 当天，摘要说明主要修改且不得包含 `/`。用户回答不明确或分支重名时继续询问，不修改仓库、不自动改名。

## 完成条件

以下条件全部满足才生成成功报告：

- 验收标准逐项核对；
- 相关检查已执行或清楚说明无法执行的原因；
- 分级要求的 Reviewer 均已返回：`fast` 为一个综合 Reviewer，`standard` 和 `strict` 为三个独立 Reviewer；
- 每个返回结果的角色与门禁要求精确匹配；`review_blocked` 或缺失任何角色时不得生成成功报告；
- 成立的高严重度问题已修复或明确列为阻塞；
- 当前 diff 只包含需求相关修改以及原有用户修改。
- `review_gate.py status` 返回 `review_phase: review_complete`，且实现完成时间不晚于首次 Review 开始时间；

群发送成功不代表代码已提交、合并或发布。报告必须明确当前代码仅存在于用户选定的本地工作位置。
